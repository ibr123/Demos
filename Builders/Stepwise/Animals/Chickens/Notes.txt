* Stepwise (Staged / Step) Builder Pattern — Summary

	What it is:
	- A builder where each method returns a DIFFERENT interface that exposes only
	  the NEXT allowed method, forcing the caller down a fixed sequence.
	- Chicken chain: Origin -> Color -> Weight -> Age -> Build.

	What problem it solves (vs the Duck recursive-generic builder):
	- The Duck builder lets you call methods in ANY order and can't require a field.
	- The Chicken builder ENFORCES order and makes every step mandatory, at COMPILE time.

	How it works:
	- IChickenOriginBuilder.Origin  returns IChickenColorBuilder
	  IChickenColorBuilder.Color     returns IChickenWeightBuilder
	  IChickenWeightBuilder.Weight    returns IChickenAgeBuilder
	  IChickenAgeBuilder.Age          returns IChickenBuilder
	  IChickenBuilder.Build           returns Chicken
	- Each interface declares only one method, so IntelliSense shows only one legal next call.
	- The concrete class returns `this` every time — it's always the same object,
	  only the static return TYPE narrows step by step.


* THE BUG I ASKED ABOUT: why could I call .Age() / .Color() / .Weight() first?

	Original entry point:
	  public static ChickenStepwiseBuilder ChickenBuilder = new();
	- It was typed as the CONCRETE class ChickenStepwiseBuilder.
	- That class implements IAggregateChickenBuilder, which exposes ALL methods.
	- So the caller saw every method and could start anywhere, e.g.
	  Chicken.ChickenBuilder.Age(2).Build();   // skipped the whole chain.

	The fix — type the entry point as the FIRST step only:
	  public static IChickenOriginBuilder ChickenBuilder => new ChickenStepwiseBuilder();
	- IChickenOriginBuilder exposes only Origin, so Origin MUST be called first.
	- Using `=>` (a property returning new) also hands out a FRESH builder each call,
	  fixing the shared-mutable-state bug of the old shared static field.

	KEY INSIGHT: a stepwise builder's enforcement depends entirely on the TYPE the
	caller is holding. Never hand the caller the concrete class or the aggregate
	interface — only ever hand out the first-step interface.


* "doesn't returning new ChickenStepwiseBuilder() leak everything, since it
   implements IAggregateChickenBuilder?"  -- THE OBJECT vs THE REFERENCE

	- At RUNTIME the object really does have all methods (it implements the aggregate).
	- But C# only lets you CALL methods visible on the COMPILE-TIME type of the
	  reference you hold.
	- The reference is typed IChickenOriginBuilder, which declares only Origin,
	  so that's the only method the compiler can see.
	- It's a narrow "view" over a fully-capable object, not a restriction on the object.

	The only leaks (both require deliberately wrong code):
	  ((IAggregateChickenBuilder)Chicken.ChickenBuilder).Age(2);   // explicit cast
	  new ChickenStepwiseBuilder().Age(2);                         // naming concrete type
	The second leak is closed by explicit interface implementation (see below).


* IS THIS POLYMORPHISM?
	  public static IChickenOriginBuilder ChickenBuilder => new ChickenStepwiseBuilder();

	- Yes — this is SUBTYPE polymorphism (a.k.a. inclusion polymorphism), the kind
	  most people mean by "polymorphism" in OOP.
	- Declared type = IChickenOriginBuilder (abstraction); actual object =
	  ChickenStepwiseBuilder (implementation). ChickenStepwiseBuilder IS-AN
	  IChickenOriginBuilder, so it's an implicit upcast.
	- The polymorphism goes "live" at the call site: calling .Origin() through the
	  interface reference dispatches to ChickenStepwiseBuilder's implementation
	  (dynamic dispatch).
	- Quick taxonomy:
	    Subtype     -> this line (interface reference to concrete object)
	    Ad-hoc      -> method overloading
	    Parametric  -> generics (the Duck recursive-generic / Enumeration<Self> builder)


* EXPLICIT INTERFACE IMPLEMENTATION — why does this signature implement the interface?
	  IChickenColorBuilder IChickenOriginBuilder.Origin(string? origin)

	Read the signature as three parts:
	    IChickenColorBuilder            = return type
	    IChickenOriginBuilder.Origin   = the qualified method name
	                                       ("the Origin that belongs to IChickenOriginBuilder")
	    (string? origin)                 = parameter

	- The interface declares:  IChickenColorBuilder Origin(string? origin);
	- The implicit form was:   public IChickenColorBuilder Origin(string? origin) { ... }
	- The explicit form just prefixes the name with the interface and drops `public`.
	  Return type and parameters are IDENTICAL.

	How C# links it:
	  1. Does the class implement IChickenOriginBuilder? (yes, via the aggregate)
	  2. Does that interface declare Origin(string?) returning IChickenColorBuilder? (yes)
	  3. Then this body IS that interface method.  -> contract satisfied.

	Two effects of the prefix:
	  - No `public` allowed: the member is NOT on the class's own public surface.
	  - Reachable ONLY through an interface reference:
	      new ChickenStepwiseBuilder().Origin("X");      // won't compile
	      ((IChickenOriginBuilder)obj).Origin("X");      // works
	  - This is exactly why we use it here: it stops callers from grabbing the
	    concrete type and calling methods out of order.


* "THE METHOD IS HIDDEN FROM THE CLASS — IT ONLY EXISTS ON THE INTERFACE" — what does that mean?

	Think of the class and its interfaces as having SEPARATE member lists:

	  IMPLICIT (normal) implementation:
	    public IChickenColorBuilder Origin(string? origin) { ... }

	    ChickenStepwiseBuilder (class surface)  |  IChickenOriginBuilder (interface surface)
	    ----------------------------------------|------------------------------------------
	    Origin(string?)  <── same method ──>  |  Origin(string?)

	    The method is added to BOTH lists. It lives on the class AND satisfies the
	    interface. You can call it through either a class reference or an interface reference.

	  EXPLICIT implementation:
	    IChickenColorBuilder IChickenOriginBuilder.Origin(string? origin) { ... }

	    ChickenStepwiseBuilder (class surface)  |  IChickenOriginBuilder (interface surface)
	    ----------------------------------------|------------------------------------------
	    [nothing — Origin is NOT here]        |  Origin(string?)  <── lives here only

	    The method is added to the INTERFACE list only. The class surface has no Origin.

	How the compiler enforces this:
	  - The compiler resolves method calls using the COMPILE-TIME TYPE of the reference,
	    not the runtime type of the object.
	  - ChickenStepwiseBuilder reference → compiler checks class surface → no Origin
	    → compile error.
	  - IChickenOriginBuilder reference → compiler checks interface surface → Origin
	    is there → allowed.

	  The object in memory is ALWAYS the same fully-capable ChickenStepwiseBuilder.
	  The difference is only which lens the compiler uses to decide what you can call.

	Analogy:
	  Think of a key card that only works on one specific door.
	  The door (interface) grants access; the building (class) does not.
	  You need the right card (reference type) to open that door.

	Implicit vs Explicit side by side:
	  // implicit — Origin is on the class, usable without the interface
	  var a = new ChickenStepwiseBuilder();
	  a.Origin("Japan");                          // ✅ compiles

	  // explicit — Origin is NOT on the class
	  var b = new ChickenStepwiseBuilder();
	  b.Origin("Japan");                          // ❌ compile error

	  IChickenOriginBuilder c = new ChickenStepwiseBuilder();
	  c.Origin("Japan");                          // ✅ only reachable via the interface


* COMPARISON: Stepwise (Chicken) vs Recursive Generic / CRTP (Duck)

	| Aspect            | Duck (CRTP)                     | Chicken (Stepwise)              |
	|-------------------|---------------------------------|---------------------------------|
	| Call order        | Any order                       | Fixed, enforced order           |
	| Skip a property?  | Yes                             | No, every step mandatory        |
	| Mechanism         | Every method returns Self       | Every method returns next iface |
	| Best for          | Optional / order-free config    | Required fields in a set order  |
	| Downside          | Can't require fields            | More boilerplate (1 iface/step) |

	- See Duck/Notes.txt last bullet: it explicitly points to the Step Builder
	  Pattern as the way to enforce call order — that's what this Chicken folder is.
