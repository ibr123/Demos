* Functional Builder Pattern — Summary
  (Dmitri Nesteruk, "Design Patterns in C# and .NET", Udemy)

	What it is:
	- A builder whose state is NOT a half-built object, but a LIST OF FUNCTIONS
	  (List<Action<Person>>) — a recipe of deferred mutations.
	- Each step (.Called, .WorksAsA) only RECORDS an action; nothing is built yet.
	- Build() creates a fresh Person and replays every recorded action onto it.

	The chain:
	  .Called("Dmitri")        -> Actions += (p => p.Name = "Dmitri")
	  .WorksAsA("Programmer")  -> Actions += (p => p.Position = "Programmer")
	  .Build()                 -> p = new Person(); run all actions; return p


* THE WHOLE POINT: Open/Closed Principle

	- PersonBuilder is `sealed` and only defines `Called`.
	- `WorksAsA` is added as an EXTENSION METHOD in a separate static class,
	  adding a NEW build step WITHOUT modifying PersonBuilder.
	- "Open for extension, closed for modification."
	- Classic builder: every new step = edit the builder class.
	  Functional builder: new step = just write an extension method.

	- THE TEST (proves OCP is the design DRIVER, not just a side benefit):
	    "If I delete the OCP benefit, does the functional design still make sense?"
	    - No. If you didn't care about third parties adding steps without
	      touching your class, you'd use a CLASSIC builder — simpler, better
	      encapsulated, and the List<Action<T>> machinery buys you nothing.
	    - The list-of-functions approach only EARNS its keep because it lets
	      extension methods plug in new steps.
	    - So every structural choice (public Actions, sealed builder, deferred
	      Build()) is bent toward making OCP extensibility possible.
	- CAVEAT: OCP is the reason behind THIS VARIANT, not the Builder pattern
	  as a whole. Builders in general exist to construct complex objects
	  step-by-step; only the functional variant is driven by OCP.


* HOW IT WORKS (mechanics)

	- Actions is `public readonly List<Action<Person>>`.
	    - readonly  = can't REASSIGN the list reference.
	    - public    = extension methods can ADD to it (needed for extensibility).
	    - contents are still mutable -> the trade-off: less encapsulation.
	- Called(name): Actions.Add(p => { p.Name = name; }); return this;
	    - the lambda CAPTURES `name` in a closure, remembered until Build().
	- Build(): var p = new Person(); Actions.ForEach(a => a(p)); return p;
	    - all deferred work runs HERE, in insertion order.
	- `return this` / `return builder` on each step enables fluent chaining.


* SEQUENCE< (full trace: from `new Person()` to invocation)

	Step 0 — Build() creates the object:
	    var p = new Person();
	    -> p.Name = null, p.Position = null   (empty shell, nothing applied yet)

	Step 1 — replay the recorded actions (Actions.ForEach(a => a(p))):
	    Actions[0]  is   p => { p.Name = name; }          // from .Called("Dmitri")
	    Actions[1]  is   p => { p.Position = position; }   // from .WorksAsA("Programmer")

	Step 2 — first invocation:  a(p)  where a == Actions[0]
	    runs  p.Name = "Dmitri";        // closure replays captured name
	    -> p.Name = "Dmitri", p.Position = null

	Step 3 — second invocation: a(p)  where a == Actions[1]
	    runs  p.Position = "Programmer"; // closure replays captured position
	    -> p.Name = "Dmitri", p.Position = "Programmer"

	Step 4 — Build() returns the now-complete p:
	    return p;   // => Person { Name = "Dmitri", Position = "Programmer" }

	Key timing note:
	    - Step 1's lambdas were RECORDED earlier, back during the chain
	      (.Called / .WorksAsA), but they only RUN now, in Build(), in order.
	    - `a(p)` = invoke the stored Action<Person> with this exact p.
	    - Same field set twice? Last invocation in the sequence wins.


* "FUNCTIONAL" — why the name?

	- Builder state = a list of functions, not object state.
	- Building = function application (apply each Action<Person> to the object).
	- That FP flavor is where "functional builder" comes from.


* CLASSIC vs FUNCTIONAL (quick table)

	| Aspect            | Classic Builder        | Functional Builder         |
	|-------------------|------------------------|----------------------------|
	| Builder state     | half-built object      | List<Action<T>>            |
	| Mutation timing   | immediately, per call  | all at once, in Build()    |
	| Add a new step    | edit the builder class | write an extension method  |
	| Encapsulation     | strong (state hidden)  | weaker (Actions is public) |
	| Best for          | fixed set of steps     | steps third parties extend |


* GOTCHAS / NOTES

	- Deferred execution: actions run in the order added; if two set the same
	  field, the LAST one added wins.
	- Closures capture the value passed at call time and replay it unchanged.
	- Repetition smell: every step is `Actions.Add(...); return ...;`.
	  -> Next lecture: GENERIC functional builder (recursive generic / CRTP)
	     with a single Do(...) helper to remove the boilerplate.
	- The course Main() never prints. Add:
	     Console.WriteLine($"{person.Name} works as a {person.Position}");
	     // Dmitri works as a Programmer


* RELATION TO OTHER BUILDER VARIANTS

	- Fluent builder   -> returns `this` for chaining (this file does that too).
	- Stepwise builder -> enforces call ORDER via interfaces (e.g. the Chicken example).
	- Faceted builder  -> multiple sub-builders for different facets.
	- Functional builder (THIS) -> optimizes for EXTENSION without modification.
	  It enforces nothing about order; its strength is open/closed extensibility.


* PROBLEM 1 — REUSED BUILDER ACCUMULATES ACTIONS (the Shape example)

	Symptom:
	- One builder instance is reused to build two products:
	    ShapeBuilder b = new();
	    Shape circle   = b.DefineShape("Circle")...Build();
	    Shape triangle = b.DefineShape("Triangle")...Build();   // SAME b
	- Build() NEVER clears ShapeBuildingActions, so the list only grows.
	  The triangle's Build() replays ALL SIX actions (circle's 3 + triangle's 3)
	  on its fresh Shape.

	Why it still looks correct:
	- Last-write-wins: the triangle's 3 actions run LAST and overwrite all three
	  fields the circle had set. Midway through Build() the object is briefly a
	  COMPLETE circle, then the triangle stamps over it.

	Where it breaks (silent bug):
	- If the triangle set Area but NOT Perimeter, the circle's leftover Perimeter
	  is never overwritten -> "triangle" silently reports the circle's perimeter.
	- Also wasted work: re-runs circle's actions every time; cost grows per product
	  (3 actions, then 6, then 9, ...).

	Root cause:
	- Builder state is a recipe that ACCUMULATES; Build() only READS it, never resets.

	Solutions (preferred -> fallback):
	  (a) BEST — a fresh builder per product. Eliminates shared state instead of
	      managing it. Match the Duck/Chicken style with a static entry point:
	          public static ShapeBuilder Builder => new();   // fresh recipe each time
	          Shape circle = Shape.Builder.DefineShape("Circle")...Build();
	  (b) FALLBACK — reset inside Build() when you MUST reuse one instance:
	          public Shape Build() {
	              Shape s = new();
	              ShapeBuildingActions.ForEach(a => a(s));
	              ShapeBuildingActions.Clear();   // next product starts clean
	              return s;
	          }
	      Downside: Build() now has a HIDDEN side effect (mutates the builder) —
	      surprising for a method that "just builds".

	Deep-understanding note:
	- Both fix the symptom. Fresh-instance ELIMINATES shared state (nothing to leak);
	  Clear() carefully MANAGES shared state (leak, then wipe). Removing a hazard
	  beats remembering to defuse it -> default to fresh.


* PROBLEM 2 — INJECTED BUILDER STAYS THE SAME INSTANCE (even when Transient)

	Symptom:
	- ShapeBuilder is injected and used to build two shapes in one class:
	    class ShapeService {
	        readonly ShapeBuilder _b;
	        ShapeService(ShapeBuilder b) => _b = b;   // resolved ONCE, stored in a field
	        ... _b.DefineShape("Circle")...Build();   // both builds share _b
	        ... _b.DefineShape("Triangle")...Build(); // -> Problem 1 is back
	    }
	- Registering AddTransient<ShapeBuilder>() does NOT help.

	Why Transient doesn't save you:
	- DI lifetime = how often the CONTAINER CREATES the instance, NOT how often you
	  reuse it.
	- Transient = "new per RESOLUTION". A constructor parameter is resolved exactly
	  ONCE (at construction), so you get one builder in a field and reuse it.
	- KEY: to get "new per USE" you must RESOLVE per use, not inject per construction.
	  No object lifetime (Transient/Scoped/Singleton) can give per-call freshness
	  through a plain injected field.

	The trap:
	- Picking Transient and thinking "new instance, so I'm safe" SHIPS the bug —
	  the safety you imagined (per-use) isn't the safety the lifetime provides
	  (per-resolution). The fix is not a different lifetime; it's changing WHAT you
	  inject so resolution happens per use.

	Solutions (preferred -> fallback):
	  (a) BEST — inject a FACTORY, not the builder. Every call resolves a fresh one:
	          class ShapeService {
	              readonly Func<ShapeBuilder> _make;
	              ShapeService(Func<ShapeBuilder> make) => _make = make;
	              ... _make().DefineShape("Circle")...Build();
	              ... _make().DefineShape("Triangle")...Build();
	          }
	      Registration (MS.Extensions.DI doesn't auto-make Func<T>):
	          services.AddTransient<ShapeBuilder>();
	          services.AddTransient<Func<ShapeBuilder>>(
	              sp => () => sp.GetRequiredService<ShapeBuilder>());
	      (Autofac and others provide Func<T> for free.)
	  (b) FALLBACK — Clear() in Build() (Problem 1's fallback) so reuse is safe
	      regardless of how the instance is shared.

	Design insight:
	- A builder is a SHORT-LIVED, SINGLE-USE, STATEFUL object. That is at odds with
	  being a long-lived injected dependency. So don't inject the builder directly
	  (that's the smell that created the question) — inject the ability to MAKE
	  builders (Func<ShapeBuilder> or a small IShapeBuilderFactory). The factory is
	  what's safe to be long-lived; each builder it hands out is fresh and single-use.
	- A static `Shape.Builder => new()` entry point also fixes Problem 1, but a
	  static new() can't be mocked/substituted, so it fights testability in a DI
	  codebase -> prefer the factory there.
