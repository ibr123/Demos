* Faceted Builder Pattern — Summary

	NAMING — this is the FACETED builder:
	- "Faceted builder" is the standard name: one product is built through several
	  separate ASPECTS (facets) — here address and job — each via its own sub-builder.
	  The project follows this name (folder Builders/FacetedBuilder, namespace
	  ...FacetedBuilder.Person, and this FacetedBuilderNotes.txt).
	- Don't confuse it with the GoF FACADE pattern — a different, unrelated pattern;
	  the name similarity is just a coincidence.
	- (Aside) The root PersonBuilder is sometimes described as a "facade" — a single
	  front door that hands out the facet sub-builders — but that's a loose description
	  of its role, not a second name for the pattern.

	What it is:
	- One product (Person) has too many properties to build with a single flat
	  fluent builder, so the properties are split into GROUPS (facets):
	      address    -> StreetName, PostCode, City
	      employment -> CompanyName, JobField, AnnualIncome
	- Each facet gets its OWN sub-builder (PersonAddressFacetBuilder, PersonJobFacetBuilder),
	  each exposing only the methods relevant to that facet.
	- A single facade (PersonBuilder) is the entry point and lets you hop between
	  facets in one fluent chain, all mutating ONE shared Person.


* THE CAST (who's who)

	- Person                    -> the PRODUCT being built (a plain data class with a ToString).
	- PersonBuilder             -> the FACADE / root. Holds the Person, exposes the facet
	                               entry points `.Address` and `.Job`, and the implicit
	                               conversion back to Person.
	- PersonAddressFacetBuilder -> facet sub-builder: .At(city, postCode, streetName)
	- PersonJobFacetBuilder     -> facet sub-builder: .At(company) .ProfessionAs(profession) .Makes(income)
	- Both facet builders INHERIT PersonBuilder (this is the load-bearing trick — see below).


* THE CHAIN (annotated — what each call returns)

	  PersonBuilder pb = new();                 // pb : PersonBuilder (facade, owns a fresh Person)
	  Person person = pb
	    .Address                                // -> new PersonAddressFacetBuilder(sharedPerson)
	      .At(city: "Amman",                    // sets City, PostCode, StreetName in ONE call;
	          postCode: "007799",               //   returns this (PersonAddressFacetBuilder)
	          streetName: "Yajooz")
	    .Job                                    // inherited entry point -> new PersonJobFacetBuilder(sharedPerson)
	      .At("Microsoft")                      // sets CompanyName;  returns this (PersonJobFacetBuilder)
	      .ProfessionAs("Developer")            // sets JobField;     returns this
	      .Makes(10);                           // sets AnnualIncome; returns this (PersonJobFacetBuilder)
	  // assignment to a `Person` variable triggers the implicit operator -> returns pb's Person
	  Console.WriteLine(person);

	Note the two facets have different granularities:
	  - Address bundles all three fields into ONE `.At(...)` call.
	  - Job chains three separate steps: `.At(...).ProfessionAs(...).Makes(...)`.
	Named arguments on `.Address.At(...)` aren't decoration — the method takes THREE
	same-typed strings, so a positional call is easy to transpose silently; naming
	pins each value to its field.


* THE KEY TRICK — FACET BUILDERS INHERIT THE FACADE (this is what makes facets switch)

	- PersonAddressFacetBuilder : PersonBuilder  and  PersonJobFacetBuilder : PersonBuilder.
	- Because each facet builder IS-A PersonBuilder, each ALSO exposes `.Address` and
	  `.Job`. THAT is the mechanism that lets you jump out of the address facet
	  (`.Address...`) straight into the job facet (`.Job...`) mid-chain.
	- Delete the inheritance and the pattern collapses: once `.Address` returned a
	  bare PersonAddressFacetBuilder, `.Job` would not exist on it and the chain
	  couldn't cross facets.

	Deep-understanding note:
	- The facets are RE-ENTERABLE and ORDER-FREE. You can bounce back and forth:
	      pb.Job.At("Microsoft").Address.At("Amman","007799","Yajooz").Job.Makes(10)   // legal
	  because every hop just returns another builder pointing at the SAME Person.
	- Trade-off of that freedom: nothing forces completeness or order. You can
	  skip a whole facet, or set a facet twice. Contrast the STEPWISE builder,
	  which uses interfaces to FORCE order/completeness. Faceted optimizes for
	  **ergonomic grouping, not for enforcement**.


* CANONICAL (INHERITANCE) vs COMPOSITION (HAS-A) — TWO WAYS TO WIRE THE FACETS

	(Note: "composition" here means HAS-A / holding a reference — NOT the GoF Composite
	pattern. Same name-collision caution as Faceted vs Facade above.)

	The facets need ONE ability: from inside one facet, jump to another (call `.Job`
	while in the address facet, or `.Address` while in the job facet). There are two
	ways to hand them that ability — this code uses the first.

	CANONICAL (what THIS code does) — inheritance, "I AM a builder too":
	      public class PersonAddressFacetBuilder : PersonBuilder   // the ": PersonBuilder"
	- Because the address builder IS-A PersonBuilder, it INHERITS the two doors
	  `.Address` and `.Job` (and the implicit `operator Person`) from the facade. You
	  write those ONCE on PersonBuilder and every facet gets them for free.
	- So the hop is ONE direct step. You never stop at a "PersonBuilder" in between —
	  the address builder already IS one, so it already carries the `.Job` door:

	      address builder ──.Job──▶ job builder      (one step, direct)

	- It works both ways (the job builder inherited the same doors), so you can bounce
	  forever:  address ──.Job──▶ job ──.Address──▶ address ──.Job──▶ job ...

	COMPOSITION — "I just HOLD the data", no ": PersonBuilder":
	      public class PersonAddressFacetBuilder                    // no base class
	      {
	          private readonly Person _person;                     // carries the shared data
	      }
	- The facet now inherits NOTHING, so you must RE-TYPE `.Address`, `.Job` and the
	  `operator Person` BY HAND on every facet. That grows fast: with N facets, each
	  facet re-declares all N doors (N x N wiring) instead of writing them once on the base.

	TRADE-OFF (in one breath):
	- Inheritance (canonical) -> least code. But a job builder isn't really "a kind of
	  person builder" (a mild lie), and the shared `person` must be left `protected` so
	  the subclasses can grab it. It's also what allowed the old
	  `new()`-instead-of-`new(person)` bug (see THE SHARED REFERENCE below).
	- Composition -> more honest, keeps `person` private, and that bug becomes a COMPILE
	  error (the ctor forces you to pass the shared Person). Price: the repetitive
	  re-wiring above.

	Bottom line: with only two facets, inheritance is the lighter, standard (canonical)
	choice — which is why this code uses it. The more facets you add, the more the
	composition re-wiring costs, and the more inheritance pays off.

	Deep-understanding note — the hop threads the SAME Person:
	- When the address builder runs its inherited `.Job` (`=> new(person)`), the `person`
	  it passes is the address builder's OWN `person` field — which is the shared one. So
	  every hop keeps handing along the SAME Person. That's exactly why it has to be
	  `new(person)` and NOT `new()`.


* THE SHARED REFERENCE — ONE Person, MANY builders (the crux of the whole pattern)

	- `protected Person person = new();` in PersonBuilder allocates the Person exactly
	  ONCE (per facade).
	- `.Address => new(person)` and `.Job => new(person)` pass THAT SAME reference into
	  each facet builder's constructor, which stores it in the inherited `person` field:
	      public PersonAddressFacetBuilder(Person person) { this.person = person; }
	  So every facet builder mutates the ONE Person, and the facets compose into one object.
	- Person is a REFERENCE TYPE, so passing it around shares the object, never copies it.

	Deep-understanding note — this is EXACTLY where the pattern silently breaks:
	- Earlier this code read `.Address => new()` (no argument). Because the facet builders
	  inherit PersonBuilder, `new()` ran the base initializer `person = new()` and handed
	  each facet its OWN fresh Person. Result: address and job wrote to DIFFERENT objects
	  and never merged — no compile error, no exception, just a half-empty Person. The fix
	  was to pass the shared `person` (`new(person)` + the ctor above). See FIELD-INITIALIZER
	  ORDERING below for WHY the inherited initializer made the bug so quiet.

	Deep-understanding note — value types would break it too:
	- If Person were a STRUCT, `new PersonAddressFacetBuilder(person)` would pass a COPY.
	  Each facet would edit its own copy, the facets would never merge, and the final
	  object would be missing whatever the other facet set. The pattern silently DEPENDS
	  on Person being a class.


* WHY PROTECTED, AND WHY PASS THE OBJECT — TWO DIFFERENT JOBS

	Passing the Person and marking the field `protected` are NOT redundant — they do
	two different jobs:
	- PASSING (`new(person)` -> ctor)  = TRANSPORT: how the shared Person gets IN.
	- `protected` on the field         = ACCESS:    permission to TOUCH it afterwards.

	Why `protected` specifically (the ctor param does NOT make it redundant):
	- The facet METHODS read the inherited field constantly, e.g. inside
	  PersonAddressFacetBuilder.At(...):
	      person.City = city;
	  `person` lives on the BASE (PersonBuilder); this method lives on the SUBCLASS, and
	  a subclass touching a base field REQUIRES `protected`.
	- If it were `private`, BOTH of these stop compiling in the facet:
	      this.person = person;   // 'PersonBuilder.person' is inaccessible (protection level)
	      person.City = city;     // same error
	- C# subtlety: a `private` base field IS inherited (it physically exists inside every
	  facet object) but is NOT accessible BY NAME from the subclass. `protected` is the
	  modifier that says "heirs may touch this."
	- You can't dodge it with a base ctor either: even if `: base(person)` did the write,
	  the At/Makes methods STILL read `person`, so it must stay `protected` regardless.

	Q: "If Person isn't passed in, do address and job stop sharing?"  ->  YES.
	- Without the passed reference each facet runs the base initializer `person = new()`
	  and gets its OWN Person; the two never merge (the old half-empty bug).
	- Sharpen one word: they don't share the same VARIABLE, they share the same OBJECT.
	  Each builder always has its OWN `person` field (its own variable); passing makes
	  all those fields POINT AT one Person:

	      passed in (shared object):
	          addressBuilder.person ─┐
	          jobBuilder.person ─────┴──▶ [ one Person ]   (address + job compose)

	      not passed (each runs person = new()):
	          addressBuilder.person ──▶ [ Person A ]
	          jobBuilder.person ─────▶ [ Person B ]        (never merge -> half-empty)

	Q: "Is `protected` what lets any inheriting class use this shared field?"  ->  YES.
	- `protected` = PersonBuilder ITSELF plus ANY subclass (the two facets, and any
	  future subclass) may read/write `person`. Unrelated/outside classes may not.

	One-line model: PASSING makes them share the same object; `protected` gives the
	subclass permission to touch that field at all. You need BOTH — no passing means no
	sharing (each gets its own object); no `protected` means the facet can't even compile
	`person.City = ...`.

	(Contrast COMPOSITION: there the facet stores the Person in its OWN `private readonly`
	field and never reaches into a base, so it CAN be private. `protected` is the rent the
	inheritance wiring charges — see CANONICAL vs COMPOSITION above.)


* THE IMPLICIT OPERATOR — the invisible Build() (there is NO .Build() here)

	  public static implicit operator Person(PersonBuilder pb) => pb.person;

	- The chain's last call `.Makes(10)` returns a PersonJobFacetBuilder, NOT a Person.
	  Yet `Person person = ...` compiles, because assigning a builder to a `Person`
	  variable fires this user-defined IMPLICIT conversion, which returns `pb.person`.
	- Since PersonJobFacetBuilder IS-A PersonBuilder, the base-typed operator applies to
	  it (and to PersonAddressFacetBuilder) too.
	- So the TERMINAL operation of the build is the assignment's type coercion — the build
	  "finishes" the moment you store it into a Person. Elegant, but subtle.

	PITFALL — `var` defeats it:
	- `var person = pb.Address.At(...)....Makes(10);` infers PersonJobFacetBuilder (no
	  target type, so no conversion). You now hold a BUILDER, not a Person.
	- Then `WriteLine(person)` prints "...Facets.PersonJobFacetBuilder" (the default
	  ToString), NOT the Person string — a silent, confusing bug.
	- The demo works precisely because it declares `Person person = ...`. The explicit
	  target type is what pulls the trigger.

	Aliasing gotcha:
	- The operator returns the SAME reference the builder holds (aliasing, not a copy).
	  If you keep `pb` and mutate it after extracting `person`
	  (e.g. `pb.Address.At("Paris", ...)`), you also mutate the already-extracted `person`.
	  A builder here is single-use by intent; treat it as spent after conversion.

	Throws on a null builder:
	- `PersonBuilder pb = null; Person p = pb;` still runs the operator, and `pb.person`
	  throws NullReferenceException. Harmless here (a null builder is nonsensical), but it
	  means this implicit conversion CAN throw — which the guideline says implicit
	  conversions shouldn't. A small point in favour of `explicit`.

	Style aside:
	- It's `implicit` (no cast) for seamless assignment. `explicit` would force `(Person)pb`.
	  Some style guides distrust implicit conversions (they hide work), but here it's the
	  deliberate, ergonomic finish line.

	Named alternate (CA2225):
	- Convention (Microsoft analyzer CA2225) is to pair a conversion operator with a
	  named method — e.g. `Build()` or `ToPerson()` — so callers in languages without
	  operator overloading can still finish. Ironic here (the point was to skip Build()),
	  but it's the standard convention.


* OPERATOR vs .Build() — TWO WAYS TO FINISH THE BUILD (why this code uses the operator)

	Both do the SAME thing — hand back the finished Person. They differ only in HOW the
	call site asks for it. This code uses the implicit operator; a plain Build() method
	is the more common, GoF-standard alternative:

	    // A) implicit operator (this code) — finish is INVISIBLE, target type drives it
	    public static implicit operator Person(PersonBuilder pb) => pb.person;
	    Person person = pb.Address.At(...).Job.Makes(10);          // no terminal call

	    // B) Build() method — finish is EXPLICIT, an ordinary terminal call
	    public Person Build() => person;
	    Person person = pb.Address.At(...).Job.Makes(10).Build();  // ends in .Build()

	Side by side:

	| Aspect               | implicit operator (this code)   | .Build() method          |
	|----------------------|---------------------------------|--------------------------|
	| Finish syntax        | invisible (target type coerces) | explicit `.Build()` call |
	| Works with `var`?    | NO — keeps builder type         | YES — infers Person      |
	| Discoverable?        | hidden (no IntelliSense)        | yes (a real method)      |
	| GoF-standard?        | no (a C# trick)                 | yes (canonical step)     |
	| CA2225 named alt.    | trips it — add a name           | it IS the named alt.     |
	| Cross-language use   | awkward (user conversion)       | trivial (plain method)   |
	| Inherited by facets? | yes (once, on the base)         | yes (once, on the base)  |

	Key trade-offs:
	- The operator's ONE selling point is the seamless finish line: `Person p = chain;`
	  with nothing tacked on the end. Build() makes you type `.Build()`, but in exchange
	  the finish is VISIBLE and self-documenting.
	- The `var` footgun is the big one: `var p = chain;` silently keeps a BUILDER with the
	  operator (no target type -> no conversion runs), whereas with Build() `var` infers
	  Person and just works. (See PITFALL in THE IMPLICIT OPERATOR above.)
	- Both are declared ONCE on the base PersonBuilder and inherited by every facet, so
	  neither wins on the cross-facet angle — that part is a tie.
	- `explicit operator` is a middle ground: `Person p = (Person)chain;` — still no named
	  method, but the visible cast forces the conversion, which defuses the `var` footgun.

	Bottom line: the operator buys a clean call site at the cost of being hidden,
	`var`-fragile, and CA2225-flagged. Build() is the safer, more conventional default.
	Plenty of real builders ship BOTH — a Build() for clarity/interop, plus an implicit
	operator for the seamless finish.


* HOW IT WORKS (mechanics, member by member)

	- PersonBuilder.person : `protected` so the facet subclasses can read AND reassign it.
	- .Address / .Job : expression-bodied GET-ONLY properties (`=>`), not fields.
	    - Each ACCESS returns a NEW facet builder via `new(person)` — cheap, all sharing
	      the one Person.
	    - Properties (not methods) so the chain reads as prose: `pb.Address.At(...)`,
	      not `pb.Address().At(...)`.
	- Facet ctors: `this.person = person;` overwrites the inherited field with the shared
	  instance handed in by the facade.
	- Facet methods: set field(s), then `return this;` -> fluent chaining WITHIN a facet.
	- Facet SWITCHING is a separate move: the inherited `.Address`/`.Job` properties,
	  which spin up a new facet builder on the same Person.

	Deep-understanding note — method-name REUSE by facet (`At` on both, on purpose):
	- BOTH facet builders define an `At(...)`:
	      .Address.At("Amman", "007799", "Yajooz")   -> City / PostCode / StreetName
	      .Job.At("Microsoft")                         -> CompanyName
	  Same verb, different type, different meaning (and even different arity) — legal
	  because they live on different facet builders. The facet acts as a NAMESPACE/CONTEXT
	  that lets you reuse a natural verb without collision. A genuine BENEFIT of splitting
	  into facets, not an accident.


* SEQUENCE (full trace: from `new PersonBuilder()` to the finished Person)

	Step 0 — new PersonBuilder():
	    PersonBuilder.person = new Person();   // one empty shell: all fields null / 0

	Step 1 — pb.Address:
	    returns new PersonAddressFacetBuilder(person)   // new facet builder, SAME person ref

	Step 2 — .At(city: "Amman", postCode: "007799", streetName: "Yajooz"):
	    person.City       = "Amman"
	    person.PostCode   = "007799"
	    person.StreetName = "Yajooz"
	    (one call sets all three address fields; returns this)

	Step 3 — .Job  (inherited property on the PersonAddressFacetBuilder):
	    returns new PersonJobFacetBuilder(person)       // yet another facet builder, SAME person

	Step 4 — .At("Microsoft") .ProfessionAs("Developer") .Makes(10):
	    person.CompanyName  = "Microsoft"
	    person.JobField     = "Developer"
	    person.AnnualIncome = 10

	Step 5 — assignment `Person person = <PersonJobFacetBuilder>`:
	    implicit operator fires -> returns pb's shared person
	    // => Person { StreetName="Yajooz", PostCode="007799", City="Amman",
	    //             CompanyName="Microsoft", JobField="Developer", AnnualIncome=10 }

	Console output (WriteLine(person) -> Person.ToString, verified by running the demo):
	    Facated Builder Example
	    StreetName: Yajooz, PostCode: 007799, City: Amman,
	    CompanyName: Microsoft, JobField: Developer, AnnualIncome: 10
	    (heads-up: the header literal reads "Facated" — a typo for "Faceted" in Person.ToString.)

	Note: the chain sets City before StreetName, but ToString prints StreetName first —
	chain/param order is independent of print order.

	Key timing note:
	    - There is no deferral here (unlike the Functional builder). Each facet call
	      mutates the shared Person IMMEDIATELY. The only thing that happens "at the end"
	      is the implicit conversion handing back that already-built object.


* WHY NOT SEALED (deliberate contrast with the Functional builder)

	- PersonBuilder is intentionally INHERITABLE — the facet builders ARE subclasses, and
	  they inherit `.Address`/`.Job` to enable facet switching. Sealing the base would
	  BREAK the pattern.
	- Compare FunctionalBuilderNotes.txt "WHY SEALED": there the extension point was
	  EXTENSION METHODS, so sealing (block subclassing) communicated the right intent.
	- One-line takeaway: extension mechanism dictates the seal decision.
	      Functional builder -> extend via extension methods -> SEAL the builder.
	      Faceted  builder   -> extend via SUBCLASS facet builders -> must stay OPEN.


* FIELD-INITIALIZER ORDERING (a real curiosity — and why the shared-ref bug was silent)

	- Constructing a facet builder allocates and immediately DISCARDS one empty Person.
	- Why: C# runs a class's field initializers as part of ITS constructor, base before
	  derived body. For `new PersonAddressFacetBuilder(person)`:
	      1. base PersonBuilder ctor runs its field initializer: person = new()
	         (an empty throwaway Person is allocated here)
	      2. PersonAddressFacetBuilder ctor body runs: this.person = person
	         (overwrites the field with the shared one; the throwaway is now garbage)
	- So the demo allocates 3 Persons total: 1 real (in `new PersonBuilder()`) + 2
	  throwaways (one per `.Address` / `.Job`). Harmless — but note the connection:
	  that base initializer is EXACTLY what silently gave each facet its own Person back
	  when the properties were `new()` instead of `new(person)`. The throwaway was meant
	  to be discarded; the bug just never replaced it with the shared one.


* FACETED vs OTHER BUILDER VARIANTS (quick table)

	| Aspect          | Faceted (THIS)             | Functional         | Stepwise            |
	|-----------------|----------------------------|--------------------|---------------------|
	| Builder state   | one shared Person, mutated | List<Action<T>>    | half-built object   |
	| Structure       | facade + subclass facets   | 1 class + plugins  | chained interfaces  |
	| Mutation timing | immediate, per call        | deferred to Build()| immediate, per call |
	| Enforces order? | NO (facets are free/reord.)| NO                 | YES (via interfaces)|
	| Finish step     | implicit operator -> Person| explicit .Build()  | explicit .Build()   |
	| Best when       | many props in clear GROUPS | 3rd-party new steps| a required sequence |


* WHEN TO USE / WHEN NOT

	- USE when a single object has many properties that fall into natural aspects,
	  and one flat fluent API would be an unwieldy wall of methods. Facets keep each
	  sub-builder small and let a verb like `At` mean different things per context.
	- DON'T reach for it when there are only a few properties (a plain fluent builder
	  is simpler), or when you must ENFORCE an order/required set (use stepwise), or
	  when third parties need to bolt on new steps without touching you (functional).


* GOTCHAS / NOTES

	- No .Build(): the terminal is the implicit conversion, driven by the TARGET type.
	  Assign to `Person` (not `var`) or you keep a builder and print the wrong thing.
	- Depends on Person being a reference type; a struct would break facet composition.
	- Facets are order-free and re-enterable; nothing guarantees completeness.
	- Builder and returned Person ALIAS the same object — the builder is spent after use.
	- `.Address.At(...)` takes three same-typed strings — use named args (the demo does)
	  to avoid silent transposition.
	- Minor: `Person.ToString()`'s header literal reads "Facated" (typo for "Faceted").


* RELATION TO OTHER BUILDER VARIANTS

	- Fluent builder   -> returns `this` for chaining (facets do this within a facet).
	- Faceted builder (THIS) -> a facade hands out multiple facet builders, one per aspect,
	  all mutating one shared product; hop between them via inherited entry points.
	- Functional builder -> state is a list of deferred actions; optimized for OCP
	  extension via extension-method plugins (see FunctionalBuilderNotes.txt).
	- Stepwise builder  -> interfaces force call ORDER (e.g. the Chicken example).
